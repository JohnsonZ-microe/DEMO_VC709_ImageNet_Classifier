 Design Checklist Instructions

Review the questions in this UltraFast Design Methodology Checklist to determine if your design or design process has any of the issues identified. Below each question is a list of suggested actions for resolving the question. In the References column, the checklist provides links and pointers to relevant information in the UltraFast Design Methodology Guide for the Vivado Design Suite and other related documentation.

The UltraFast Design Methodology Guide provides higher level recommendations and directives. The User Guides and Tutorials provide more instructional information on how to operate the tools to support the methodology.

Editing the Status, Verbosity Level, and Visibility of Checklist Questions

  • The first column in the checklist displays and allows you to edit the status of individual items.
  • To modify the status or the verbosity level of an individual question or all the questions in a section or subsection, click the status icon.
  • To hide or show a group of questions, click the expand/collapse button to the right of the section title.

Adding, Editing, and Displaying User Notes

  • To add User Notes to a question, click the status icon for that question.
  • To highlight all User Notes in red, set the user preference in the Settings Dialog available from the main toolbar.
  • To show or hide the User Notes for individual questions, click the stauts icon and edit the verbosity level.

Saving, Exporting, and Importing Design Checklists

  • The Design Checklist automatically saves changes, but you need to manually save backups or additional copies as needed.
  • To export a checklist attach the XCHK file and send it via email to the desired recipient.
  • To import a checklist, right-click in the tree view and select "Import Design Checklist...".
  • To modify the Project Overiew or Project Notes for a specific checklist, right-click the checklist name in the tree view and select "Edit Design Checklist Properties...".

1.4.2 - Design Migration 
 
Are you planning to migrate your design forward as new releases of Vivado are introduced?

Action: Xilinx recommends always migrating designs and IP to new releases, as they become available. There are rules on which IP versions can be used with which software release.

In order to configure any IP, it must be done in the software release that the IP version is available in. In other words a previous version of IP cannot be re-configured in future software releases. However, previous versions of IP can be used but they are locked with the current configuration.

Updating IP can sometimes require logic updates to the design, so using locked IP may be desired at times.

2.2.1 - Pin Out 
 
Have you defined the pin out within a project in the Vivado IDE?

Have you used the IP generation tools to help select the pinouts for Memory and GT based IP?

Have you put 'prohibits' on needed Vref pins to avoid assigning to other I/Os to them?

Have you set the required device properties?

Have you run report_drc and report_ssn on the fully defined IO project?

Action: In order to implement the design and to generate a bitstream, a target part must be selected and all physical I/O constraints need to be defined. The I/O planning features of the Vivado IDE can help you assign and validate the I/O constraints. The Vivado IDE has a comprehensive I/O planning environment that facilitates I/O planning at various stages of the design process. A Pin Planning project is used to facilitate early I/O planning prior to having top-level RTL. An RTL project is used after the top-level RTL has been created. Defining or validating the I/O pinout after synthesis is recommended, as the netlist is now clock aware.

The required information to provide is:

I/O locations
Direction of I/Os
Part compatibility
Configuration method
I/O standard
Drive Strength
Slew Rates
Pullups/pulldowns
DCI cascade

Device properties like configuration method, CFGBVS etc. enable greater DRC checking.

Vivado has the report_drc and report_ssn commands to help you validate the I/O pinout and to perform signal noise analysis.

2.2.2 - SSI Technology 
 
Are you using stacked silicon interconnect (SSI) technology?

Have you planned the logic intended for each SLR, including clocks, GTs, and memory interfaces?

Action: SSI technology requires special attention during I/O planning. Familiarize yourself with the SSI technology requirements and recommendations.

To optimize performance when designing with SSI technology devices, partition the design using floorplanning techniques to target specific SLRs, keeping the utilization of each SLR within guidelines. Attempt to reduce the number of critical signals crossing SLR boundaries. This also involves proper planning of the I/O interfaces, clocks and logic associated with the floorplan.

If possible, with a floorplan in mind, ascertain if crossing SLRs in the SSI device is a bottleneck.

If you are using one or more Memory Controller IPs, have you analyzed the placement of the Memory Interfaces to avoid overly congested areas?

This is especially important when using devices with I/O columns through the center of the device, such as with some of the UltraScale devices.

3.2.4 - Constraint Creation 
 
Are your XDC constraints compliant with the recommended guidelines?

Do they accurately reflect clocking on the top level?

Are constraints created in a manner that will not affect other parts of the design once integrated?

Action: During the SubModule design phase, a dummy top level constraints file can provide clocks to the SubModule. At this stage, inter clock paths may not be correctly defined. Ensure that invalid inter clock paths are correctly constrained using report_clocks and report_clock_interaction and report_timing Tcl commands.

At this stage, clock constraints will not accurately model skew. Improve accuracy of sub module timing by including a full clock network back to source pins.

When submodules are implemented standalone, care must be taken with definition of cells to ensure that after integration with the top level, extra cells are not included in exceptions.

3.2.6 - Post Synthesis Performance Estimation 
 
Have you validated the constraints coverage and timing for synthesized design?

Does the synthesized design meet timing?

Action: It is important to perform post-synthesis timing analysis after development of each sub-block to ensure proper constraints coverage and design performance. If the design is not close to meeting timing after synthesis, it likely will not after implementation either. Timing failures should be fixed on the front end through RTL and constraints changes.

To validate timing, open the synthesized design and run the following commands to analyze the results:

1) Run report_timing or report_timing_summary.
2) Run report_clock_interaction.
3) Run report_design_analysis.
4) Run report_cdc.

4.2.2 - Resets 
 
Does the reset functionality follow Xilinx guidelines?

Does the design have a reset that guarantees a safe startup?

Action: You should: minimize the number of resets; reset control logic only, and avoid resetting the datapath; avoid active low resets. Run the report_reset_signals.tcl custom Tcl script to ensure proper use of resets.

Global resets are often not necessary in FPGA design and can consume large amounts of routing and can cause timing issues both inside and outside the reset timing paths. It is suggested to only code resets in the portions of the design that are necessary for proper operation and avoid resets in all other areas of the design.

Resets should be synchronized in their own clock domain.

Consider keeping key parts of the design held in reset until all clocks are stable.

4.4.4 - Routing Issues 
 
Does the design exhibit routing congestion?

Is the router having difficulties fixing hold time violations?

Action: Routing congestion can be the cause of differences in timing estimates between a placed design and a routed design. It is best identified in the early stages of routing in the log file. Level 5 or above is concerning. More detailed analysis can be carried out using report_design_analysis.

Synthesis options such as LUT combining, MUXFx optimization, carry chain length and SRL optimization can drastically influence routing congestion. Alternatively, you can use opt_design to reduce MUXF*/CARRY*/SRL* in the modules located in the congested areas. You can also remove the LUT combining properties before place_design. Use report_qor_suggestions to automatically generate these suggestions using the Vivado tools.

Vivado also has implementation strategies that are specifically targeted at reducing routing congestion. You can reduce congestion on congested modules using the CELL_BLOAT_FACTOR property.

Use the phys_opt_design hold fixing options to reduce the number of hold violations before routing. These options also help to reduce congestion.

4.4.5 - Implementation Options 
 
Have you tried a number of design run strategies to improve runtime and implementation consistency?

Have you done a thorough analysis for each implementation run?

Are implementation options used to improve runtime and consistency?

Action: Vivado allows previous DCP checkpoint files to be used as part of an incremental flow when minimal changes have been made. This helps improve design consistency and can help runtime when minor changes to the database have been made.

Vivado allows parallelism through out of context synthesis and implementation. Are BDs and IPs synthesized in out of context fashion? This is the simplist way to improve run time. Further enhancements can be made by using the bottom up flow for other RTL blocks.

Vivado allows post implementation database edits. In particular, this can be used to modify debug probes for the ILA and connections to debug IOs.

Consistency can be achieved by locking down hard blocks such as block RAM and DSP slices from a previous run. This can improve the number of strategies meeting timing on each run.

Vivado allows usage up to 8 cores. On Windows machines, is this set to the maximum for the machine in use? The default for Linux is 8.

Low effort changes:
Run power_opt_design.
Switch to lower power speed grade.
Switching off clocks when not required.

Higher effort changes:
High level restructuring of the design.
Recoding of RTL to control enable signals and use power efficient features in FPGA like power down on RAMs.
